# Global programming rules

Rules are grouped by scope. Apply **Global** to all work. Apply **Enterprise** and **Personal** only when the project clearly fits that category — if unsure, ask.

## Global (applies to everything)

- **Type safety.** Use types wherever the language supports them, including in dynamically-typed languages. Prefer explicit types on function signatures and data structures over loose/untyped containers. If a type is genuinely unknown, make that explicit rather than implicit.
- **Pragmatism, but type safety counts.** Don't overengineer — abstractions, layers, and patterns must earn their keep. But constructs that *look* like ceremony (newtypes, value classes around primitives, sum types instead of boolean flags) often earn it by making invalid states unrepresentable. Judge by whether the type system gets stronger, not by surface line count.
- **Compile-time invariants over runtime validation + tests.** If a constraint can be expressed in the type system so invalid states won't compile, do that instead of adding runtime checks and tests to enforce it. Runtime validation is the fallback for things types genuinely can't express (external input, dynamic data, true runtime conditions).
- **Prefer libraries over hand-rolled plumbing.** For functionality outside the core of the application (date parsing, HTTP, retries, validation, etc.), reach for a maintained library rather than writing your own utilities. Hand-rolled code is for *the actual business logic* — don't reinvent what the ecosystem already solves well.
- **Business rules live on entities/values, not services.** Follow the *tell, don't ask* principle: behavior belongs on the object that owns the data. Prefer methods on entity/value classes over generic decoupled service classes that read state and decide externally. Reach for a service only when logic genuinely spans multiple aggregates or needs external collaborators.
- **Immutable by default; contained mutation inside domain objects.** Default to immutable data — produce new values rather than mutate in place. Inside domain entities, mutation is allowed but must go through named behavior methods (`order.markAsShipped()`, `invoice.applyDiscount(...)`), never raw setters. Services should not sequence chains of setters to drive an object's state — if you're reaching for that, the behavior belongs on the entity.
- **Task runner: `just`.** Use [`just`](https://github.com/casey/just) (a `justfile`) for project task automation. Don't introduce `make`, npm scripts as the primary runner, shell-script grab-bags, or other task runners when adding new automation.
- **Construct objects via factories / named-parameter constructors.** Object creation should encapsulate its own build rules. Use factory methods (`User.register(...)`, `Order.fromCart(...)`) or constructors with named parameters that require all mandatory fields. Never pass `null`/`None` to a constructor to "skip" a field — that's a signal you need a different factory method that takes only what's actually required for that case. Each factory should validate invariants up-front so an object cannot exist in an invalid state.
- **No mapper services.** Object conversion belongs on one of the two types involved, not in a separate `XMapper`/`XConverter` service. Use a `from*` constructor/factory (`UserDto.fromEntity(user)`) or a `to*` method (`user.toDto()`) — whichever respects the dependency direction (the type that *may depend on* the other owns the conversion). This keeps conversion rules next to the type that defines them and avoids a parallel hierarchy of mapper classes.
- **Validation lives on value objects / entities.** Invariants are enforced at construction so an invalid instance cannot exist. Don't validate in DTOs and then "trust" the domain — let the domain types refuse bad input themselves. This pairs with the factory rule: factories run the checks.
- **Comments: short, direct, one line.** Comments are welcome but must be concise — one line, stating the *why* or the non-obvious fact. No multi-line essays, no restating what the code does, no boilerplate docstrings that just echo the signature.
- **Domain language in names.** Use business vocabulary, not technical jargon: `Order.cancel()` not `OrderManager.processStatusUpdate()`. Avoid generic suffixes like `*Manager`, `*Helper`, `*Util`, `*Processor` — they're a sign the name isn't pulling its weight. Class and method names should read like the domain expert would phrase them.
- **Sync by default, async only when it pays.** Don't reach for async/await/non-blocking code by reflex. Use it for I/O-bound paths that genuinely benefit (concurrent fan-out, high-throughput services). Synchronous code is easier to reason about, debug, and test — keep it that way unless there's a concrete reason.
- **Conventional Commits.** All commit messages follow [Conventional Commits](https://www.conventionalcommits.org/): `type(scope): subject`, e.g. `feat(billing): add invoice export`, `fix(auth): reject expired tokens`, `refactor:`, `docs:`, `test:`, `chore:`. Use `!` or a `BREAKING CHANGE:` footer for breaking changes. Applies to every repo — enterprise and personal.
- **No interfaces unless the scenario demands one.** Interfaces (or abstract base classes) with a single implementation are noise — delete the interface and use the concrete class. Even when there *are* multiple implementations, consider whether a sum type with pattern matching / `switch` on a discriminator is clearer than dynamic dispatch. Reach for an interface only when you genuinely need a seam (external boundary, true plugin point, or test double that can't be replaced more simply).
- **Read operations return absence, never throw.** Methods that look up or fetch data must not throw when nothing is found. Return `null` / `Option` / `Optional` (whichever the language uses) for single-item lookups, and an empty collection for queries. Exceptions are for genuine errors (I/O failure, programming bugs), not for "the row didn't exist."
- **Don't extract functions just to name steps.** Sequential code that runs top-to-bottom is easier to read than a chain of one-call-deep helpers that force the reader to jump around. Extract a function only when there's actual reuse, or when the same task has multiple implementations (e.g. provider/strategy variants). "It reads nicer as small functions" is not a sufficient reason.
- **Negate with a `not(...)` helper, not `!`.** The `!` operator is too easy to misread — a single character flips meaning. In languages where `!` is the negation operator (JS/TS, Java, C#, Kotlin, etc.), define a small `not(x)` helper and use it consistently. Languages with a `not` keyword (Python, Ruby) are fine as-is.

## Enterprise / company applications

- **Follow existing conventions first.** In an established codebase it's often impractical to apply every best practice — surrounding code, frameworks, and team agreements constrain what's reasonable. Match the conventions already in the file/module/project before reaching for the rules below. The rest of this section describes preferences for *new* code or greenfield areas; don't refactor existing code to match them unless asked.
- **No primitive obsession.** Avoid raw primitives (`string`, `int`, `UUID`, etc.) for domain concepts. Wrap them in newtypes / value classes (e.g. `UserId`, `EmailAddress`, `Money`) so the type system carries meaning and invariants are enforced at construction. A function signature should tell you *what* a value is, not just its underlying representation.
- **Mapping external payloads.** When parsing payloads from external systems (HTTP responses, message queues, third-party APIs):
  1. **Map only the fields you actually use.** Don't mirror the entire upstream schema "for completeness" — unused fields are dead weight that breaks when the upstream changes shape.
  2. **Enums as strings, not native enums.** External vocabularies evolve — new values get added, old ones removed. Parsing into a closed enum will break at runtime the moment that happens. Keep the value as a string and convert only when your code branches on it (and handle unknown values explicitly).
  3. **Nested structs as inner classes.** When a payload contains sub-structures, define them as inner/nested classes of the top-level payload type, in the same file. Don't scatter one-file-per-DTO across the project for transient external shapes — they belong with the payload they're part of.
- **Co-locate things that are read together.** Input/output types, file-scoped constants, small helper classes — if you only ever look at them alongside their owner (a use case, a payload, an entity), they live in the same file. Don't fragment for the sake of "one class per file."
- **Reach for domain verbs over operators and conditional chains.** Prefer named helpers that read like sentences over raw operators or inline conditionals: `toggle.disabled()` over `!toggle.isEnabled`, `order.canBeCancelled()` over a multi-clause boolean expression at the call site. The cost is a tiny helper; the gain is code that reads like the domain expert would phrase it.
- **Module layout: layered, with the domain at the core.** Preferred top-level structure:
  - `app/rest`, `app/<protocol>`, `app/listeners` — transport adapters (HTTP, gRPC, message consumers, CLI, etc.).
  - `domain/entities`, `domain/services`, `domain/usecases` — business model and orchestration.
  - `integration/` — repositories and outbound clients (DB, third-party HTTP, queues).
  - `shared/` — cross-cutting concerns: logging, metrics, security, cloud SDK wrappers (AWS, etc.).
  Domain code can call into `integration/` directly — no ports/adapters indirection required. Apply to greenfield projects only; if an existing project uses a different layout, follow it and don't restructure to match this.
- **Extract magic numbers/strings to named constants.** Inline literals lose meaning quickly in business code. Give them names so the *intent* shows up in the call site, not just the value.
- **Structured logging throughout, with discipline.** Log across layers when there's real signal, but always structured (key/value pairs, not interpolated strings) and always with context (correlation/trace IDs, user/tenant where relevant). Tune levels carefully — `info` is for things an operator would want to see; debug noise stays at `debug`. No `System.out.println`-style logs.
- **WireMock-in-Docker is acceptable for external dependencies.** Tests don't need to hit real third-party services. Standing up [WireMock](https://wiremock.org/) (or equivalent HTTP stubbing) as a container alongside the test environment is a fine substitute — it gives realistic HTTP behavior without the flakiness, cost, or rate limits of the real service. Prefer this over in-process HTTP-client mocks when the integration shape matters.
- **Use cases orchestrate, not controllers/listeners/services.** Each application action should have a dedicated *use case* (interactor / application service) that orchestrates the business rules end-to-end. Controllers, HTTP handlers, message listeners, and CLI entry points should be thin: parse input, invoke the use case, format output. Don't chain services-calling-services to compose a workflow, and don't push orchestration into transport-layer code.
- **The use case entry point should have a legible shape.** The main execution method of a use case (e.g. `execute`, `handle`) should read as a sequence of phases — typically *load inputs → validate → compute → execute → persist* — visually separated (blank lines or short comments) so a reader can scan the structure before reading details. Consistent shape across use case entry points beats clever flow inside any one of them. This applies only to the entry point, not to private helpers on the same class.
- **Tests should be short.** Each test should read quickly: arrange-act-assert in a handful of lines. Push repetitive setup into helper functions/fixtures so the test body shows intent, not plumbing. If a test needs heavy setup even with helpers, treat that as a smell — the code under test is probably reaching for the wrong abstraction (too many collaborators, leaky boundaries, missing aggregate). Fix the design, don't grow the test.

## Personal scripts

- **No tests required.** Personal projects/scripts don't need unit, integration, or e2e tests. Don't add a test suite, test files, or test scaffolding unless I explicitly ask. Manual verification is fine.
- **Magic numbers/strings are usually fine inline.** Don't extract constants just to give a literal a name in a one-off script. Extract only when the value is genuinely reused or its meaning is unclear in context.
